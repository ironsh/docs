# Overview

A **coding agent** is a long-running VM that runs an AI coding harness like Claude Code against a specified GitHub repository. Agents run inside iron.sh VMs, but add a layer of workflow tooling on top: a persistent `tmux` session, automatic repository cloning, credential injection, and a built-in TUI for monitoring egress traffic. Agents are managed through the `irons agents` command group.

## Creating and Attaching to an Agent

To create a new agent, run `irons agents new` with a GitHub repository:

```sh
irons agents new --repo github.com/myorg/myapp
```

This provisions a VM, clones the repo, injects your secrets, and drops you into the agent TUI automatically.

To attach to an agent that's already running:

```sh
irons agents attach <agent-name>
```

The agent name defaults to the repository name (e.g. `myapp`). This can be customized by passing `--name` when creating the agent. You can list your running agents with `irons agents list`.

## The Agent TUI

When you attach to an agent with `irons agents attach`, you get a terminal UI with two panes:

- **Agent Pane** — the `tmux` session where the coding agent is running. This is where you interact with Claude Code (or whatever agent you're using). It works exactly like a normal terminal session.
- **Egress Monitor** — a live view of every outbound connection the VM is making, showing the destination host, request count, allow/deny status, and when it was last seen.

You flip between the two panes with `Ctrl-]`.

### Egress Monitor Controls

The egress monitor is an interactive TUI, not just a log tail. You can navigate and take action directly:

| Key       | Action                                          |
| --------- | ----------------------------------------------- |
| `↑` / `↓` | Navigate between hosts                          |
| `a`       | Allow the selected host (add an egress rule)    |
| `d`       | Deny the selected host (remove its egress rule) |
| `i`       | Show detail view for the selected host          |
| `Ctrl-]`  | Switch back to the agent pane                   |
| `q`       | Exit the TUI                                    |

The status bar at the bottom of the screen shows a running count of allowed and denied connections, and highlights the most recently denied host so you can spot issues at a glance.

## Agents vs. VMs

A VM is a general-purpose compute environment with egress controls. You SSH in, do your work, and tear it down. An agent is an opinionated wrapper around a VM, purpose-built for AI coding workflows:

|                         | VM                                        | Agent                                          |
| ----------------------- | ----------------------------------------- | ---------------------------------------------- |
| **Provisioned with**    | `irons create`                            | `irons agents create`                          |
| **Tied to a repo**      | No                                        | Yes — clones the repo on creation              |
| **Credentials**         | Manual SSH or `irons ssh`                 | Injected automatically from your secrets store |
| **Session persistence** | None — each SSH connection is independent | `tmux` session that survives disconnects       |
| **Interaction model**   | Raw SSH                                   | TUI with split agent / egress monitor views    |
| **Intended for**        | Any workload                              | AI coding agents (Claude Code, etc.)           |

Under the hood, an agent _is_ a VM. Everything that applies to VMs — egress rules, snapshots, secret proxying, audit logging — applies to agents too. The agent layer just makes the common workflow faster.

## How It Works

When you run `irons agents new`, Iron.sh:

1. Provisions a VM
2. Clones the specified GitHub repository into the VM
3. Injects credentials from your [secrets store](/reference/secret-proxying) as environment variables
4. Starts a `tmux` session with your chosen coding agent (e.g. Claude Code)
5. Begins recording egress traffic, and drops you into the TUI

The agent keeps running whether or not you're connected. Exiting the TUI (`Ctrl-c` from the agent pane, or `q` from the egress monitor) leaves the agent running in the background. You can re-attach at any time without interrupting the agent's work.

## Credential Injection

Agents automatically inject secrets from your Iron.sh secrets store into the VM as environment variables. If you've already configured secrets with `irons secrets add`, they're available inside the agent session without any extra setup.

For example, if you have a `GITHUB_TOKEN` secret and an `ANTHROPIC_API_KEY` secret, both will be present as environment variables when the agent starts. The egress proxy handles token replacement transparently — see [Secret Proxying](/reference/secret-proxying) for details.
